## 一、自注册工厂模式

### 1.1 完整设计

```cpp
// 1. 工厂类（单例）
class StrategyFactory {
    using Creator = std::function<std::shared_ptr<AIStrategy>()>;
    unordered_map<string, Creator> creators;

    static StrategyFactory& instance() {
        static StrategyFactory factory;  // Meyers' Singleton
        return factory;
    }
};

// 2. 自注册辅助模板
template<typename T>
struct StrategyRegister {
    StrategyRegister(const string& name) {
        StrategyFactory::instance().registerStrategy(name, []{ return make_shared<T>(); });
    }
};

// 3. 各策略文件中一行注册
static StrategyRegister<AliyunStrategy> reg1("1");
```

### 1.2 为什么用单例

工厂是全局唯一的注册表，Meyers' Singleton 保证线程安全的延迟初始化（C++11 标准保证）。

### 1.3 为什么叫"自注册"

利用 C++ **全局/静态变量在 main() 之前构造**的特性：静态变量构造函数执行时自动向工厂注册，不需要手动写 if/else 或 switch。

### 1.4 新增模型：只需两步

1. 继承 `AIStrategy`，实现虚函数
2. 在 `.cpp` 文件中加一行：`static StrategyRegister<NewStrategy> reg("5");`

完全不修改工厂代码 —— **开闭原则（OCP）**。

## 三、数据库连接池

### 3.1 为什么用 queue

FIFO 确保连接被均匀使用。如果用 LIFO（栈），某些连接被反复使用，其他长期空闲可能被数据库超时断开。

### 3.2 get（获取连接）

```cpp
shared_ptr<DbConnection> getConnection() {
    unique_lock<mutex> lock(mutex_);
    cv_.wait(lock, [this]{ return !connections_.empty(); });
    auto conn = connections_.front();
    connections_.pop();
    return conn;
}
```

### 3.3 自定义删除器（归还而非销毁）

正常 shared_ptr 引用计数归零时 delete，通过自定义删除器将 delete 替换为**把连接放回池中**：

```cpp
return shared_ptr<DbConnection>(raw_conn, [this](DbConnection* p) {
    unique_lock<mutex> lock(mutex_);
    connections_.push(shared_ptr<DbConnection>(p, [](DbConnection*){}));
    cv_.notify_one();
});
```

### 3.4 心跳检测

后台线程每 30 秒遍历所有空闲连接，执行 `mysql_ping`，失效则重连。O(n) 遍历但连接数通常只有 5-20 个。

## 四、RabbitMQ 异步入库

### 4.1 不能规避所有慢 SQL

MQ 只是把延迟从前台转移到后台。消费者仍会被慢 SQL 阻塞导致消息堆积。

### 4.2 读取一致性

**"读己之写"问题解决方案**：
- 前台：同步写入内存 messages[] → 立即可见
- 后台：MQ → MySQL → 异步持久化
- 实时读走内存，历史恢复走数据库

### 4.3 上锁 vs 异步

Handler 线程只等待 MQ publish（~0.1-1ms），慢的部分在后台线程。锁只保护 AMQP channel 线程安全，不涉及 MySQL。

## 五、系统设计

### 5.1 HTTP 对象池

```cpp
template<typename T>
class ObjectPool {
    std::queue<unique_ptr<T>> pool_;
    unique_ptr<T, function<void(T*)>> acquire();  // 复用或新建
    void release(T* obj);  // reset 后归还或销毁
};
```

### 5.2 HTTP/1.1 vs HTTP/2

| 特性 | HTTP/1.1 | HTTP/2 |
|------|----------|--------|
| 传输格式 | 文本 | 二进制帧 |
| 多路复用 | 不支持 | 单连接多流 |
| 头部压缩 | 无 | HPACK |
| 队头阻塞 | 有 | 无 |

升级需要：二进制帧解析器、流管理、HPACK 编解码器、ALPN 协商、流量控制。建议引入 nghttp2 库。

### 5.3 AI API 异步化

- 方案 1：线程池 + future
- 方案 2：curl_multi + EventLoop 集成
- 方案 3：SSE + 异步回调

### 5.4 Nginx 负载均衡 + Redis Session

多实例部署时 Session 存内存会导致登录失效 → 迁移到 Redis 共享。

### 5.5 限流（令牌桶）& 熔断（Closed → Open → Half-Open）

限流：限制请求速率，返回 429。
熔断：下游故障时快速失败，返回降级响应。

### 5.6 线程安全

已保护：chatInformation、onlineUsers、连接池、MQ、Session。
优化方向：读写锁（shared_mutex）、细粒度锁、无锁数据结构、原子操作。
