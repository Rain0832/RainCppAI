# v1.3.0 优化小结：异步线程池处理 AI 调用

> 对应 TODO 优化 1 · 大改动

---

## 一、问题背景

muduo 的请求处理流程是同步的：

```
IO线程: onMessage → onRequest → httpCallback_(req, &resp) → conn->send(resp)
                                      ↑ 这里调 Handler::handle()
                                      ↑ Handler 内 curl_easy_perform() 阻塞 500ms~数秒
                                      ↑ 整个 IO 线程被卡住
```

4 个 IO 线程 = 最多 4 个并发 AI 请求。第 5 个请求必须等前面的完成。

## 二、解决方案：线程池 + 异步回写

### 2.1 架构变化

```
Before:
  IO线程 → Handler::handle() → curl阻塞 → 填resp → conn->send()
           ~~~~~~~~ 全程阻塞 IO 线程 ~~~~~~~~

After:
  IO线程 → Handler::handle() → 提交任务到线程池 → 立即返回（deferred）
           ~~~~~~~~ IO 线程空闲，可处理其他请求 ~~~~~~~~
                                    ↓
  线程池线程 → curl_easy_perform() → 拿到结果 → runInLoop → conn->send()
           ~~~~~~~~ 在独立线程中阻塞，不影响 IO ~~~~~~~~
```

### 2.2 关键改动

#### HttpResponse 异步模式

```cpp
class HttpResponse {
    bool deferred_ = false;              // 延迟发送标记
    TcpConnectionPtr conn_;              // 持有连接引用

    void setDeferred(bool on);           // Handler 设置
    bool isDeferred() const;
    void setConnection(const TcpConnectionPtr& conn);
    TcpConnectionPtr getConnection() const;
};
```

#### HttpServer::onRequest 条件发送

```cpp
void HttpServer::onRequest(const TcpConnectionPtr& conn, const HttpRequest& req) {
    HttpResponse response(close);
    response.setConnection(conn);         // 注入连接
    httpCallback_(req, &response);

    if (response.isDeferred()) return;    // 异步模式：不发送，Handler 自行处理

    // 同步模式：照旧发送
    Buffer buf;
    response.appendToBuffer(&buf);
    conn->send(&buf);
}
```

#### Handler 异步提交

```cpp
void ChatSendHandler::handle(const HttpRequest& req, HttpResponse* resp) {
    // 1. 同步部分：鉴权、参数解析、获取 AIHelperPtr（读写锁）
    // ...

    // 2. 标记异步
    resp->setDeferred(true);
    auto conn = resp->getConnection();

    // 3. 提交到线程池
    server_->aiThreadPool_.submit([conn, AIHelperPtr, ...]() {
        std::string answer = AIHelperPtr->chat(...);  // 在线程池中阻塞

        // 4. 通过 runInLoop 回到 IO 线程发送响应
        conn->getLoop()->runInLoop([conn, answer]() {
            conn->send(buildHttpResponse(answer));
        });
    });
    // handle() 立即返回，IO 线程空闲
}
```

#### ThreadPool 实现

```cpp
class ThreadPool {
    vector<thread> workers_;
    queue<function<void()>> tasks_;
    mutex mutex_;
    condition_variable cv_;

    template<typename F, typename... Args>
    auto submit(F&& f, Args&&... args) -> future<invoke_result_t<F, Args...>>;
};
```

### 2.3 线程安全保证

| 操作 | 线程安全机制 |
|------|-------------|
| `conn->send()` | 通过 `runInLoop` 投递到 IO 线程执行 |
| `TcpConnectionPtr` | `shared_ptr`，线程池持有防止连接提前释放 |
| `conn->connected()` | 发送前检查连接是否仍存活 |
| `chatInformation` | `shared_mutex` 读写锁（v1.2.0 已完成） |

## 三、性能对比

| 指标 | Before | After |
|------|--------|-------|
| 并发 AI 请求 | 4（= IO 线程数） | 8（= 线程池大小，可配置） |
| IO 线程阻塞 | 是（AI 调用期间） | 否 |
| 新请求响应 | 排队等待 | 立即处理（鉴权/参数解析） |
| 同步请求影响 | 无 | 无（GET /chat 等同步接口不受影响） |

## 五、面试考点

- **为什么用 runInLoop 而不是直接 conn->send()？** muduo 的 TcpConnection 不是线程安全的，所有 IO 操作必须在其所属的 EventLoop 线程中执行。`runInLoop` 将回调投递到正确的 IO 线程
- **TcpConnectionPtr 生命周期？** `shared_ptr`，线程池 lambda 捕获后引用计数 +1，即使客户端断开连接，对象也不会提前释放（但 `connected()` 返回 false）
- **为什么不用 curl_multi？** curl_multi 需要深度集成 EventLoop 的 fd 监听，复杂度远高于线程池方案，且不具备通用性
- **线程池大小如何选择？** AI API 调用是 IO 密集型（等待网络），8-16 线程即可。CPU 密集型任务建议等于核心数
