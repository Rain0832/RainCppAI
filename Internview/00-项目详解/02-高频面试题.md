## A. 项目介绍（必问）

**Q1: 请介绍一下你的项目**

> 我做的是一个 **C++17 实现的 AI 应用服务平台**。整个项目分为两层：
>
> **底层是自研的 HTTP 服务器框架**（HttpServer），基于 muduo 网络库的 Reactor 模型，支持 HTTP 解析（状态机）、路由系统（精确 + 正则双匹配）、中间件链（CORS 等）、Session 会话管理、MySQL 连接池、SSL/TLS 加密通信。
>
> **上层是 AI 聊天应用**（ChatServer），支持：
> - 多模型 LLM 对话：策略模式 + 工厂模式 + 静态自注册，一键切换通义千问、豆包等
> - RAG 检索增强：对接阿里百炼知识库
> - MCP 工具调用：两段式推理，模型自主判断是否调用工具
> - 图像识别：ONNX Runtime + MobileNetV2
> - 语音合成/识别：百度 TTS/ASR
> - 异步入库：RabbitMQ 将聊天记录异步写入 MySQL

**Q2: 为什么选 C++？**

> 性能天花板高（muduo epoll Reactor）；学习深度（手写 HTTP 框架）；ONNX Runtime C++ API 性能最优；展示全栈能力。

## B. 网络编程 & HTTP 框架

**Q3: Reactor 模型？**

> 主线程 accept，Round-Robin 分发到 N 个子线程 EventLoop（One Loop Per Thread）。每个 EventLoop 通过 `epoll_wait` 监听 fd 事件，触发回调：`onMessage → 状态机解析 → 路由 → Handler → 发送响应`。

**Q4: HTTP 解析器怎么实现？**

> 有限状态机：`kExpectRequestLine → kExpectHeaders → kExpectBody → kGotAll`。解决 TCP 粘包/拆包问题。

**Q5: 路由设计？**

> 双策略：精确匹配 `unordered_map` O(1)，正则匹配 `vector<regex>` O(n)。先精确后正则。

**Q6: 中间件设计？**

> 洋葱模型：`before₁ → before₂ → Handler → after₂ → after₁`。已实现 CorsMiddleware。

**Q7: 异步响应怎么实现？（v1.3.0）**

> `HttpResponse::setDeferred(true)` 注入 `TcpConnectionPtr`。Handler 提交任务到线程池，`onRequest` 检测到 deferred 不发送。线程池完成后 `conn->getLoop()->runInLoop(send)`，保证 IO 线程安全。

## C. AI Agent & 大模型

**Q8: 什么是 RAG？**

> 调用 LLM 前先检索相关文档注入 Prompt。优化方向：分块、向量数据库、混合检索、Reranking、Query 改写。

**Q9: 什么是 MCP？工具调用怎么实现？**

> 两段式推理：① 意图识别（把工具列表注入 Prompt，让 LLM 判断是否调用）→ ② 结果综合（工具结果回注 Prompt）。v1.6.0 目标：升级为标准 JSON-RPC 2.0 MCP Server。

**Q10: 策略模式怎么用？**

> `AIStrategy` 基类 + 4 个实现 + `StrategyRegister<T>` 静态自注册（利用 C++ 静态变量 main() 前初始化特性）。

**Q11: 多会话管理？**

> 二级 Map：`unordered_map<userId, unordered_map<sessionId, AIHelper>>`，`shared_mutex` 读写锁，LRU 淘汰（MAX_SESSIONS=500）。

## D. 基础架构 & 中间件

**Q12: 数据库连接池？**

> 单例 + `queue` + `mutex` + `condition_variable` + 自定义删除器归还连接 + 后台心跳检测 + `wait_for(3s)` 超时保护。

**Q13: RabbitMQ 异步入库？**

> Handler → `MQManager::publish`（~1ms）→ RabbitMQ → 2 个消费线程执行 INSERT。`exclusive=false` 支持多消费者。

**Q14: SSL/TLS？**

> 自定义 BIO 回调桥接 muduo 非阻塞 IO 和 OpenSSL 阻塞 IO。

**Q15: AIHelper 线程安全？（v1.4.0）**

> `msgMutex_` 保护 `messages_` 向量；`atomic<bool> processing_` CAS 保证同一 session 串行（第二条消息快速失败而非阻塞）；`Message.role` 显式字段消除奇偶下标依赖。

## E. C++ 高频八股

**Q16: shared_ptr vs unique_ptr？** 共享 vs 独占所有权。`shared_ptr` 引用计数，适合多所有者；`unique_ptr` 零开销，适合单所有者传递。

**Q17: 可变参数模板？** 参数包 + 递归展开，编译期类型安全。项目中用于 `DbConnection::executeQuery/executeUpdate` 绑定 SQL 参数。

**Q18: runInLoop vs queueInLoop？** 条件异步 vs 始终异步的跨线程任务投递。项目中线程池完成 AI 调用后用 `runInLoop` 安全回写 conn。

**Q19: RAII？** 构造获取 + 析构释放。项目中 `shared_ptr<DbConnection>` 自定义删除器归还连接池，`ProcessingGuard` 确保 `processing_` 解锁。

**Q20: std::move？** `static_cast<T&&>`，移动语义真正发生在移动构造/赋值中。

## F. 系统设计 & 性能

**Q21: 支持 10W QPS？**

> 网络层（SO_REUSEPORT 多实例）+ 协议层（对象池）+ 业务层（异步 AI + Redis 缓存）+ 架构层（Nginx + 限流熔断）。

**Q22: 如何压测当前 QPS？**

> `wrk -t4 -c100 -d30s http://127.0.0.1:8088/` 测静态页面基线；带 Cookie 的 Lua 脚本测 `/chat/history`；关注 P99 延迟和错误率。

**Q23: 三高表设计体现在哪？（v1.5.0）**

> `messages.id` AUTO_INCREMENT（高可用）；`role ENUM` + 追加写（高并发）；`sessions.deleted_at` 软删除 + `DATETIME(3)` 毫秒精度（高扩展）。

## 复习要点速记

- [ ] 五层架构：网络→协议→中间件→路由→业务
- [ ] Reactor：One Loop Per Thread + epoll + 非阻塞 IO
- [ ] 状态机：kExpectRequestLine → kExpectHeaders → kExpectBody → kGotAll
- [ ] 双匹配路由：精确 O(1) + 正则 O(n)
- [ ] 洋葱中间件：before 顺序 → Handler → after 逆序
- [ ] 策略+工厂+自注册：开闭原则，静态变量 main 前初始化
- [ ] MCP 两段式推理：意图识别 → 工具调用 → 结果综合
- [ ] 异步响应：deferred + runInLoop，线程池不阻塞 IO 线程
- [ ] 连接池：queue + mutex + cv + wait_for(3s)超时 + 自定义删除器
- [ ] MQ：exclusive=false 多消费者，削峰填谷，至少一次语义
- [ ] AIHelper：msgMutex_ + processing_ CAS 串行化
- [ ] Message.role 显式，消除奇偶下标
- [ ] shared_mutex：shared_lock 读 / unique_lock 写
- [ ] LRU：list + unordered_map，O(1)
- [ ] 三高表：AUTO_INCREMENT PK + role ENUM + deleted_at + DATETIME(3)
