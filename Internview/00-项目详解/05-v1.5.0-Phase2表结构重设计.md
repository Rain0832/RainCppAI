# v1.5.0 优化小结：数据库表结构重设计（Phase 2）

> 中等改动 · 三高可扩展表结构

---

## 一、背景

旧 `chat_message` 表存在以下问题：

| 问题 | 影响 |
|------|------|
| `id` 字段复用为 `user_id`，无消息自增主键 | 无法精确定位单条消息 |
| `is_user` 字段为 TINYINT，角色信息需要业务代码计算 | 不支持 system prompt 等扩展角色 |
| 会话仅存内存 `sessionsIdsMap`，无持久化 | 服务重启后会话列表为空，需要重新从消息表推断 |
| 无 API Key 持久化，用户每次刷新浏览器需重新输入 | 体验差 |

## 二、新表结构

### sessions 表（新增）
```sql
CREATE TABLE sessions (
  id         VARCHAR(64) NOT NULL PRIMARY KEY,  -- sessionId (UUID)
  user_id    BIGINT UNSIGNED NOT NULL,
  title      VARCHAR(128) DEFAULT NULL,          -- AI 生成的标题
  model_type VARCHAR(32) DEFAULT NULL,
  created_at DATETIME(3) NOT NULL DEFAULT CURRENT_TIMESTAMP(3),
  updated_at DATETIME(3) NOT NULL DEFAULT CURRENT_TIMESTAMP(3) ON UPDATE CURRENT_TIMESTAMP(3),
  deleted_at DATETIME DEFAULT NULL,             -- 软删除，支持回收站
  INDEX idx_user_id (user_id),
  INDEX idx_updated (updated_at)
);
```

### messages 表（替代 chat_message）
```sql
CREATE TABLE messages (
  id         BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY,  -- 真正的消息序号
  session_id VARCHAR(64) NOT NULL,
  user_id    BIGINT UNSIGNED NOT NULL,
  role       ENUM('user','assistant','system') NOT NULL,  -- 显式角色
  content    MEDIUMTEXT NOT NULL,
  model      VARCHAR(64) DEFAULT NULL,                     -- 哪个模型回复的
  created_at DATETIME(3) NOT NULL DEFAULT CURRENT_TIMESTAMP(3),
  INDEX idx_session_created (session_id, created_at),
  INDEX idx_user_id (user_id)
) ROW_FORMAT=DYNAMIC;
```

### user_api_keys 表（新增）
```sql
CREATE TABLE user_api_keys (
  id         BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
  user_id    BIGINT UNSIGNED NOT NULL,
  provider   VARCHAR(32) NOT NULL,   -- 'aliyun' / 'doubao' / 'baidu'
  api_key    VARCHAR(512) NOT NULL,  -- 可加密存储
  created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
  updated_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  UNIQUE KEY uk_user_provider (user_id, provider)
);
```

## 三、代码改动

### readDataFromMySQL
```cpp
// 旧：SELECT is_user → 奇偶推断 role
SELECT is_user, content, ts FROM chat_message ORDER BY ts ASC

// 新：直接读 role 字段
SELECT m.session_id, m.user_id, m.role, m.content,
       UNIX_TIMESTAMP(m.created_at)*1000 AS ts_ms
FROM messages m
INNER JOIN sessions s ON m.session_id = s.id
ORDER BY m.created_at ASC, m.id ASC
```

### pushMessageToMysql
```cpp
// 旧：写 chat_message，用 ts 毫秒时间戳
INSERT INTO chat_message (id, username, session_id, is_user, content, ts) VALUES (...)

// 新：INSERT IGNORE sessions（幂等创建会话）+ INSERT messages
INSERT IGNORE INTO sessions (id, user_id) VALUES ('...', 123);
INSERT INTO messages (session_id, user_id, role, content) VALUES ('...', 123, 'user', '...');
```

### ChatSessionsHandler
- 从 `sessions` 表读取 `title`，展示 AI 生成的会话标题
- 降级策略：DB 查询失败时使用内存列表

## 四、三高设计体现

| 维度 | 设计 |
|------|------|
| **高可用** | `deleted_at` 软删除，数据不丢失；`messages.id` 自增主键，消息唯一可寻址 |
| **高并发** | 消息写入通过 RabbitMQ 异步，`messages` 表仅追加写入（无 UPDATE），锁竞争极低 |
| **高扩展** | `role` ENUM 可扩展 system/tool；`model` 字段记录每条回复的模型；`sessions.title` 支持 AI 智能命名 |

## 五、数据迁移

旧数据通过以下 SQL 迁移：
```sql
-- sessions
INSERT IGNORE INTO sessions (id, user_id, created_at, updated_at)
SELECT DISTINCT session_id, id AS user_id, ... FROM chat_message GROUP BY session_id, id;

-- messages
INSERT INTO messages (session_id, user_id, role, content, created_at)
SELECT session_id, id, IF(is_user=1,'user','assistant'), content, FROM_UNIXTIME(ts/1000, ...)
FROM chat_message ORDER BY ts ASC;
```

旧 `chat_message` 表保留不删除，作为备份。

## 六、面试考点

- **为什么用 DATETIME(3) 而不是 BIGINT 时间戳？** DATETIME(3) 可读性强，支持毫秒精度，MySQL 原生时区处理，`INDEX` 效率与 BIGINT 相当
- **role ENUM 还是 VARCHAR？** ENUM 存储只占 1-2 字节（vs VARCHAR 的 4+ 字节），且有值域约束；已知角色集合有限时优先 ENUM
- **sessions.deleted_at 软删除的意义？** 硬删除不可恢复，软删除支持"回收站"功能、数据审计、误删恢复
- **messages.id AUTO_INCREMENT 主键的意义？** 保证唯一可寻址，支持分页查询（`WHERE id > last_id`）、精确定位单条消息、彻底消除奇偶排序依赖
