## v2.0.0 — Agent Loop + 原生 Function Calling

**核心改动**：`AIHelper::agentLoop()` 实现 N 步 Function Calling 循环（最多 5 步），替代旧的两段式 Prompt 注入推理

**架构变化**：
```
旧（两段式）：
  user → Prompt注入工具列表 → LLM(意图) → 解析自由文本JSON → 工具 → LLM(综合) → answer

新（Agent Loop）：
  user → LLM(带tools[]声明) → [tool_calls?]
                                ├─ yes → 执行工具 → 结果回注 → LLM → [tool_calls?] → ...循环
                                └─ no  → answer
```

**关键代码**：
- `AIToolRegistry::getToolsSchema()` — 返回 OpenAI 兼容 `tools[]` 格式
- `agentLoop()` — 解析响应中 `tool_calls` 结构化字段（非 Prompt 注入）
- `buildRequest(messages, tools)` — 所有 Strategy 支持 `tools` 参数
- `chatStream()` MCP 模式 — Agent Loop 后逐字 UTF-8 安全回调

**面试考点**：Prompt 注入 vs 原生 Function Calling（可靠性 vs 兼容性）| Agent Loop 防死循环（maxSteps=5）| tool 角色消息入上下文但当前不持久化（TODO）

## v1.6.1 — Phase A 紧急 Bug 修复

4 项修复：chatStream MCP 分支、豆包 Endpoint ID、历史消息 DB fallback、天气工具超时+降级

**面试考点**：SSE 模式下 MCP 被跳过的根因（chatStream 无 MCP 分支）| 降级策略（网络失败返回模拟数据而非崩溃）

**SSE**：`AIHelper::chatStream()` + `StreamWriteCallback` 解析 OpenAI 兼容 SSE 格式，逐 token 回调 → `ChatSseHandler` 发送 `text/event-stream` 头 → 线程池执行 → `runInLoop` 逐 token 推送

**MCP**：`McpServer` 实现 `tools/list` + `tools/call`（JSON-RPC 2.0），复用 `AIToolRegistry`，可被 Claude Desktop/Cursor 直接接入

**面试考点**：SSE vs WebSocket（单向 vs 双向）| fetch streaming vs EventSource（POST vs GET）| curl 回调在线程池线程，通过 runInLoop 回 IO 线程

## v1.5.0 — 三高表结构

**改动**：新建 `sessions`（UUID PK, title, deleted_at 软删除）+ `messages`（AUTO_INCREMENT PK, role ENUM）+ `user_api_keys`（user_id+provider UK）

**面试考点**：DATETIME(3) vs BIGINT | ENUM vs VARCHAR 存储效率 | 软删除意义 | AUTO_INCREMENT 消除奇偶依赖

## v1.4.0 — 并发 Bug 修复

6 项修复：
1. `AIHelper::messages_` 加 `msgMutex_`（无锁 → 有锁）
2. `Message.role` 显式字段（消除奇偶下标）
3. `atomic<bool> processing_` CAS 串行化（同一 session 并发保护）
4. `GetMessages()` 锁范围修复（chatInfo 锁 vs msgMutex_ 职责分离）
5. MQ `exclusive=false`（多消费线程）
6. DB `wait_for(3s)`（超时快速失败）

**面试考点**：为什么用 CAS 而非 mutex（lock-free 快速失败）| 锁职责分离原则 | exclusive 队列语义

---

## v1.3.0 — 异步线程池

**改动**：`ThreadPool` 类 + `HttpResponse::setDeferred()` + `onRequest` 条件发送 + Handler 提交到线程池 + `runInLoop` 异步回写

**面试考点**：为什么用 runInLoop（TcpConnection 非线程安全）| TcpConnectionPtr 生命周期（shared_ptr 引用计数）| 线程池大小选择（IO 密集 8-16）

---

## v1.2.0 — 读写锁 + LRU

**改动**：`std::mutex` → `std::shared_mutex`，读用 `shared_lock`，写用 `unique_lock`。LRU 用 `list + unordered_map` O(1) 淘汰。

**面试考点**：shared_mutex mutable 声明 | double-check 必要性 | LRU 为什么用 list+map
