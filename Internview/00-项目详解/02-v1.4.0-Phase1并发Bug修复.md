# v1.4.0 优化小结：并发 Bug 修复（Phase 1）

> 中等改动 · 修复 6 项并发安全缺陷

---

## 一、背景

代码审查发现系统存在多处数据竞争（UB）和逻辑错误，在高并发或用户快速操作时必然触发。

---

## 二、修复详情

### 2.1 AIHelper::messages 加锁

**问题**：`messages_` 是普通 `vector`，无任何同步原语。多个线程池任务持有同一 `AIHelper` 实例时（用户对同一 session 并发请求），并发 `push_back` 是未定义行为。

**修复**：
```cpp
class AIHelper {
    mutable std::mutex   msgMutex_;   // 新增
    std::vector<Message> messages_;   // 所有读写均加锁
};

// addMessage、GetMessages、restoreMessage 均：
std::lock_guard<std::mutex> lock(msgMutex_);
```

---

### 2.2 消除奇偶下标判断角色

**问题**：原设计 `messages[0]=user, [1]=AI, [2]=user, ...`，依赖偶数=user、奇数=AI 的脆弱约定。用户连发两条消息时：

```
messages: [user1]  ← 第一条入队
           [user2] ← 用户第二条消息并发 push（AI还没回）
           [AI1]   ← AI1 回来后 push
```

此时 `messages[1]` 是 user2 但被当作 AI 渲染，所有后续消息角色全部错位。

**修复**：引入 `Message` 结构体，显式存储 `role`：

```cpp
struct Message {
    std::string role;    // "user" 或 "assistant"
    std::string content;
    long long   ts;
};

// ChatHistoryHandler 直接读 role：
msgJson["is_user"] = (msg.role == "user");  // 不再用 i % 2
```

---

### 2.3 同一 Session 并发串行化

**问题**：同一 session 两条并发请求会同时操作 `messages_`，即使加了锁，`chat()` 的 addMessage-curl-addMessage 序列也会被打断，逻辑乱序。

**修复**：`atomic<bool> processing_` + CAS 保证同一时刻只有一个 `chat()` 在执行：

```cpp
std::atomic<bool> processing_; // 初始化为 false

// chat() 入口：
bool expected = false;
if (!processing_.compare_exchange_strong(expected, true)) {
    return "[提示] 当前会话正在处理上一条消息，请稍后再试";
}
// RAII 解锁
struct Guard { std::atomic<bool>& f; ~Guard() { f = false; } } g{processing_};
```

> **为什么不用 mutex？** atomic CAS 是 lock-free 的，不会阻塞调用线程（直接返回错误），比 mutex 阻塞等待更适合"快速失败"场景。

---

### 2.4 GetMessages() 锁范围修复

**问题**：原代码在 `chatInfo` 写锁释放后调用 `GetMessages()`：

```cpp
{
    unique_lock wlock(chatInfo);
    AIHelperPtr = ...;
}  // ← 写锁在此释放
messages = AIHelperPtr->GetMessages();  // ← 此处无任何保护！
// 与此同时线程池里的 chat() 正在 push_back messages_
```

**修复**：chatInfo 锁只保护 map 查找（职责单一），消息读取由 AIHelper 自身的 `msgMutex_` 保护：

```cpp
// chatInfo 锁完全释放后：
std::vector<Message> messages = AIHelperPtr->GetMessages();
// GetMessages() 内部：lock_guard(msgMutex_); return messages_;
```

---

### 2.5 RabbitMQ 独占队列 Bug

**问题**：`DeclareQueue(queue_name_, false, true, false, false)` 第三参数 `exclusive=true`。独占队列只能被声明它的连接访问——第一个消费线程声明后，第二个线程尝试访问同一队列时收到 `ACCESS_REFUSED (403)`，线程异常退出，实际只有一个消费者在工作。

**修复**：
```cpp
// exclusive: true → false
channel->DeclareQueue(queue_name_, false, false, false, false);
```

---

### 2.6 DB 连接池等待超时

**问题**：`cv_.wait(lock)` 无超时。8 个线程池任务同时发起 AI 请求，每个任务都需要 DB 连接（写消息），但连接池只有 5 个。第 6-8 个任务永久阻塞在等待，耗尽线程池资源。

**修复**：
```cpp
bool acquired = cv_.wait_for(lock, std::chrono::seconds(3), [this] {
    return !connections_.empty();
});
if (!acquired) {
    throw DbException("Connection pool timeout: no available connection within 3s");
}
```

## 三、面试考点

| 问题 | 核心答案 |
|------|---------|
| 为什么用 atomic 而不是 mutex 做 session 串行化？ | CAS 是 lock-free，第二条消息直接返回错误不阻塞，避免线程池任务堆积 |
| Message.role 显式字段的好处？ | 不依赖存储顺序，支持 system prompt、tool response 等新角色，扩展性强 |
| chatInfo 锁和 msgMutex_ 的职责分离原则是什么？ | 单一职责：chatInfo 锁只管 map 的增删，AIHelper 的 msgMutex_ 只管消息向量 |
| RabbitMQ exclusive=true 的作用和副作用？ | 作用：队列与连接生命周期绑定，连接断开自动删除。副作用：同一队列只能被一个连接访问，多消费者必须 exclusive=false |
| 连接池等待超时的意义？ | 防止雪崩：数据库慢或宕机时，线程池任务快速失败并返回错误，不会耗尽线程池资源 |
