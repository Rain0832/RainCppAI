## 一、架构深挖

### Q1：AIHelper → curl 线程池是同步还是异步？

**本质：对 IO 线程是异步，对线程池线程是同步。** 这叫"异步代理"模式。

```
IO 线程:  Handler → setDeferred(true) → submit(task) → 立即返回
                                              ↓ (异步边界)
线程池:   task() → chat() → curl_easy_perform() 阻塞 1~5s → runInLoop → conn->send()
                    ↑ 这段是同步阻塞的
```

**refactor 体现**（v1.3.0 → v2.0）：
- v1.3.0 前：`chat()` 在 IO 线程中 `curl_easy_perform` 阻塞，4 IO 线程 = 最多 4 并发 AI 请求
- v1.3.0 后：引入 `deferred + ThreadPool + runInLoop` 三件套
- v2.0 后：Agent Loop 也在同一线程池中同步执行，`chatStream` 的 MCP 模式走非流式 Loop 后逐字回调

**代码关键**（`ChatSendHandler.cpp:86-102`）：
```cpp
resp->setDeferred(true);  // 告诉 HttpServer "别替我发响应了"
auto conn = resp->getConnection();
server_->aiThreadPool_.submit([conn, AIHelperPtr, ...]() {
    std::string answer = AIHelperPtr->chat(...);  // 同步阻塞
    sendAsyncResponse(conn, ...);  // runInLoop 回 IO 线程
});
```

**面试话术**：> "线程池对 IO 线程是异步的——Handler 提交任务后立即返回，IO 线程不阻塞。但线程池内部 curl_easy_perform 是同步阻塞的，这是 IO 密集型任务，阻塞等网络响应是合理的。关键是把阻塞从 IO 线程移到了独立线程池。"

### Q2：SSE 流式有什么实现难点？不是只改前端就行吗？

**不是**，后端有三个核心难点：

| 难点 | 说明 | 代码位置 |
|------|------|---------|
| **1. TcpConnection 线程安全** | curl WRITEFUNCTION 在线程池线程回调，但 `conn->send()` 只能在 IO 线程调用。每个 token 必须 `runInLoop` | `ChatSseHandler.cpp:8-10` |
| **2. SSE 协议握手顺序** | 必须先发 `text/event-stream` 头，才能逐 token 推送。握手必须在 IO 线程的 `runInLoop` 中 | `ChatSseHandler.cpp:73-83` |
| **3. 连接生命周期** | 用户关闭页面后 AI 还在输出。每次 `onChunk` 前 `conn->connected()` 检查，否则写已关闭连接会崩溃 | `ChatSseHandler.cpp:92` |

**如果只改前端**：前端 `fetch().body.getReader()` 确实能读流式数据，但后端不配合的话：
- 不发 `text/event-stream` 头 → 浏览器不按 SSE 协议解析
- 不用 `runInLoop` → 线程池线程直接 `conn->send()` → 竞态崩溃
- 不检查 `connected()` → 客户端断开后往死连接写 → 内存泄漏/崩溃

**面试话术**：> "SSE 难点不在前端，在后端的线程安全。curl 回调在线程池线程，conn->send 只能在 IO 线程。所以每个 token 都要 runInLoop 回 IO 线程发送，还要处理客户端中途断开的生命周期问题。"

### Q3：MCP 和 Function Calling 的关系？项目怎么用的？

**两者是不同层级的概念**：

```
Function Calling = LLM API 的参数格式（tools[] 字段）
MCP              = 通信协议（Client ↔ Server，JSON-RPC 2.0）

关系：MCP Client 内部可以用 Function Calling 让 LLM 决定调哪个工具
      MCP 是 Function Calling 之上的通信协议层
```

| 维度 | Function Calling | MCP |
|------|-----------------|-----|
| 本质 | API 参数格式 | 通信协议 |
| 谁执行工具 | 客户端自己写代码执行 | MCP Server 执行 |
| 工具来源 | 每个函数自己写逻辑 | 可复用任何 MCP Server 暴露的接口 |
| 层级 | API 层 | 协议层 |
| 独立性 | 绑定 LLM API 格式 | 独立于任何 LLM |

**项目中两者都用了**：
- **Function Calling**：`agentLoop()` → `buildRequest(messages, tools)` 声明 `tools[]` → LLM 返回 `tool_calls` → 解析执行
- **MCP Server**：`McpServer` 类 → JSON-RPC 2.0 `tools/list` + `tools/call` → 让 Claude Desktop/Cursor 调用我们的工具

**面试话术**：> "Function Calling 是告诉 LLM 我有什么工具的 API 格式，MCP 是 LLM 应用和工具提供方之间如何通信的协议标准。项目中 Agent Loop 内部用 Function Calling，对外通过 MCP Server 暴露工具。MCP 更高级，因为它是 Function Calling 之上的协议层。"

### Q4：MySQL 连接池怎样定义的？有什么问题？

**当前实现**（`DbConnectionPool`）：

```
初始化：创建 N 个连接放入 queue<shared_ptr<DbConnection>>
获取：  mutex + cv.wait_for(3s) + queue.front() + pop
归还：  shared_ptr 自定义删除器，析构时 push 回 queue + cv.notify_one()
心跳：  后台线程每 60s 遍历所有空闲连接 ping
```

**自定义删除器核心代码**（`DbConnectionPool.cpp:88-93`）：
```cpp
return shared_ptr<DbConnection>(conn.get(), 
    [this, conn](DbConnection*) {
        lock_guard<mutex> lock(mutex_);
        connections_.push(conn);    // 归还而非 delete
        cv_.notify_one();           // 唤醒等待者
    });
```

**有什么问题**：

| 问题 | 说明 |
|------|------|
| 心跳拷贝整个 queue | `auto temp = connections_` 拷贝队列，连接数多时浪费 |
| `detach()` 线程 | 无法优雅关闭，程序退出可能中断 ping |
| 只检查空闲连接 | 正在使用的连接断开只能到归还时 `ping()` 才发现 |
| 连接泄漏风险 | `reconnect()` 失败时异常路径可能丢失连接 |

### Q5：CAS 原理？什么时候用 CAS vs mutex？

**CAS（Compare-And-Swap）原理**：

CPU 指令级原子操作：如果当前值 == 期望值，就设为新值，否则不做修改。

```cpp
// AIHelper.cpp:147 — 具体代码
bool expected = false;
if (!processing_.compare_exchange_strong(expected, true))
    return "[提示] 当前会话正在处理上一条消息";
// 语义：如果 processing_ == false → 设为 true 并返回 true
//       如果 processing_ == true  → 不修改，返回 false
```

**CAS vs Mutex 选择**：

| 场景 | 选 CAS | 选 Mutex |
|------|--------|---------|
| 竞争少，快速失败 | ✅ `processing_` 串行化 | ❌ 不需要等锁 |
| 需要等待/重试 | ❌ CAS 会自旋浪费 CPU | ✅ mutex + cv 让出 CPU |
| 保护复杂操作 | ❌ CAS 无法保护多步操作 | ✅ mutex 保护临界区 |
| 简单状态标记 | ✅ atomic\<bool/int\> | ❌ 杀鸡用牛刀 |

**项目具体体现**：

| 数据 | 锁策略 | 原因 |
|------|--------|------|
| `processing_` | atomic CAS | 快速失败，不等锁，适合"同一 session 只处理一条消息" |
| `messages_` | mutex | push_back 是多步操作，需要临界区保护 |
| `chatInformation` | shared_mutex | 读多写少，允许并行读 |
| `connections_` (连接池) | mutex + cv | 需要等待连接归还，cv 让出 CPU |

**面试话术**：> "CAS 是 CPU 级原子操作，适合竞争少、快速失败的场景。项目中 processing_ 用 CAS，因为同一 session 并发请求只需要知道'有人在用'然后快速返回，不需要排队等待。messages_ 用 mutex 因为需要保护多步操作。选择原则是：简单状态标记用 CAS，复杂临界区用 mutex，读多写少用 shared_mutex。"

### Q7：架构优缺点？改进空间？能学到什么？

**优点**：
1. 五层分层清晰，HTTP 框架可独立复用
2. 异步线程池 + deferred 是 muduo 生态标准实践
3. 策略 + 自注册工厂，开闭原则执行到位
4. 并发安全体系完整（读写锁 + CAS + LRU）

**缺点与改进**：

| 缺点 | 改进方向 | 参考项目 |
|------|---------|---------|
| Handler 碎片化 + friend 滥用 | 引入 Context/Service 层解耦 | Go Gin Context |
| 业务和 HTTP 框架耦合 | 注解式/声明式序列化 | Spring Boot @ResponseBody |
| 单机架构，水平扩展难 | Redis 共享 + 无状态设计 | Nginx + Redis Session |
| 缺乏 Service/Repository 层 | Handler → Service → Repository 三层 | MVC 分层 |
| 全局读写锁 | 分片锁 / concurrent_map | Java ConcurrentHashMap |

**从这个项目能学到什么**：
1. **Reactor 模式**：One Loop Per Thread 的工程实践
2. **异步编程**：deferred + runInLoop 的线程间通信范式
3. **并发设计**：锁选型（CAS vs mutex vs shared_mutex）的实战判断
4. **设计模式**：策略 + 工厂 + RAII 在真实项目中的运用
5. **架构演进**：从 v1.0 同步到 v2.0 Agent Loop 的迭代思路

**进阶学习路线**：
- muduo（陈硕）→ Reactor + 线程模型
- Go Gin → 路由分组 + 中间件 + Context 设计
- brpc（百度）→ C++ 工业级 RPC 框架
- Spring Boot → IoC + 自动配置 + 分层架构
