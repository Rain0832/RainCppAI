## 二、面试自我介绍（2分钟版）

> 我做的是一个 **C++17 AI 应用服务平台**，分为两层：
>
> **底层是自研 HTTP 框架**，基于 muduo 的 Reactor 模型，实现了 HTTP 状态机解析、双匹配路由（精确 O(1) + 正则 O(n)）、中间件洋葱模型、会话管理、SSL/TLS，以及异步线程池——AI 请求不阻塞 IO 线程。
>
> **上层是 AI 聊天应用**，核心特性包括：
> - **Agent Loop**：N 步 Function Calling 循环，支持多步工具调用
> - **4 种大模型热切换**：策略模式 + 自注册工厂，新增模型只需两行代码
> - **SSE 流式输出**：curl 回调逐 token 推送，前端实时渲染
> - **标准 MCP Server**：JSON-RPC 2.0 实现 tools/list + tools/call
> - **RAG 知识库**：对接阿里百炼，检索增强生成
> - **并发安全**：shared_mutex 读写锁、atomic CAS 串行化、LRU 淘汰
> - **异步入库**：RabbitMQ 削峰，消息队列写 MySQL
>
> 项目从 v1.0 迭代到 v2.0，修复了 6 项并发 Bug，重设计了三高表结构，实现了 Agent Loop 和真正的流式输出。

## 三、高频追问速答

### A. 项目 & 架构

| 问题 | 答案 |
|------|------|
| 为什么用 C++？ | 性能天花板高 + 深入学习网络/并发 + ONNX C++ API 最优 |
| 项目分几层？ | 五层：网络(muduo) → 协议(状态机) → 中间件(洋葱) → 路由(双匹配) → 业务(Handler) |
| 最大性能瓶颈？ | AI API ~1-5s 阻塞。解决：异步线程池(8线程) + deferred 响应，IO 线程零阻塞 |
| 架构缺点？ | Handler friend 滥用、全局读写锁、缺 Service 层。改进：Context 层解耦、分片锁、三层分离 |

### B. 网络

| 问题 | 答案 |
|------|------|
| Reactor 模型？ | One Loop Per Thread，主线程 accept，Round-Robin 分发到 N 个子线程 EventLoop |
| HTTP 解析？ | 有限状态机 4 状态，解决 TCP 粘包/拆包 |
| 异步响应？ | setDeferred(true) + 线程池 + runInLoop 回写 |
| SSE 实现？ | curl WRITEFUNCTION 逐 chunk 回调 + text/event-stream + fetch streaming |
| SSE 难点？ | 不是只改前端：①线程池线程不能直接 conn->send ②SSE 握手顺序 ③连接生命周期 |

### C. AI Agent

| 问题 | 答案 |
|------|------|
| Agent Loop？ | N 步 Function Calling 循环：LLM 返回 tool_calls → 执行工具 → 结果回注 → 继续调用 LLM，直到不再调工具 |
| MCP vs FC？ | MCP 是通信协议（Client-Server），FC 是 API 参数格式。MCP 更高级，是 FC 之上的协议层 |
| MCP 怎么实现？ | 标准 JSON-RPC 2.0 Server：tools/list + tools/call，复用 AIToolRegistry |
| RAG？ | 检索增强生成：对接阿里百炼知识库 API，ragId 通过前端配置传入 |
| 策略模式？ | AIStrategy 基类 + 4 实现 + StrategyRegister 静态自注册，新增模型零修改工厂代码 |

### D. 并发安全

| 问题 | 答案 |
|------|------|
| CAS 原理？ | CPU 原子操作：比较当前值==期望值则设为新值。适合简单状态标记、快速失败场景 |
| CAS vs Mutex？ | CAS：竞争少、快速失败、简单标记；Mutex：需等待/重试、复杂临界区 |
| 锁策略？ | chatInfo 用 shared_mutex（读多写少）；messages_ 用 mutex（多步操作）；processing_ 用 CAS（快速失败） |
| 用户连发消息？ | atomic<bool> CAS 串行化，第二条直接返回"请稍后再试"（非阻塞快速失败） |
| DB 连接池？ | queue + mutex + cv.wait_for(3s) + 自定义删除器归还 |
| LRU 淘汰？ | list + unordered_map，O(1) touch/evict，MAX 500 |

### E. 数据库

| 问题 | 答案 |
|------|------|
| 表设计？ | sessions(软删除) + messages(AUTO_INCREMENT PK + role ENUM) + user_api_keys |
| 为什么改？ | 旧表用奇偶下标判断角色，连发消息必错位；新表 role ENUM 显式存储 |
| 异步入库？ | MQ publish ~0.1ms → RabbitMQ → 2 消费线程 INSERT。内存优先，DB 异步 |
| 连接池问题？ | 心跳拷贝整个 queue、detach 线程无法优雅关闭、只检查空闲连接、reconnect 失败可能泄漏 |

### F. C++ 八股

| 问题 | 答案 |
|------|------|
| shared_ptr vs unique_ptr？ | 共享 vs 独占。项目中 AIStrategy 共享，SslContext 独占 |
| RAII？ | 构造获取 + 析构释放。ProcessingGuard 确保 processing_ 解锁 |
| move？ | static_cast\<T&&\>，移动在移动构造/赋值中 |
| runInLoop？ | 线程间安全任务投递。当前线程直接执行，否则 queue + wakeup |

## 五、复习 Checklist

- [ ] 五层架构：网络→协议→中间件→路由→业务
- [ ] Reactor：One Loop Per Thread + epoll
- [ ] 状态机：4 状态解决粘包/拆包
- [ ] 路由：精确 O(1) + 正则 O(n)
- [ ] 洋葱中间件：before → Handler → after
- [ ] 策略+工厂+自注册：开闭原则
- [ ] 异步响应：deferred + runInLoop（对 IO 异步，对线程池同步）
- [ ] SSE 三难点：线程安全 + 握手顺序 + 连接生命周期
- [ ] Agent Loop：N 步 Function Calling，tool_calls 结构化解析
- [ ] MCP vs FC：MCP 是协议层，FC 是 API 格式，MCP 更高级
- [ ] CAS：CPU 原子操作，适合快速失败；mutex 适合临界区；shared_mutex 适合读多写少
- [ ] 消息安全：msgMutex_ + CAS + Message.role
- [ ] LRU：list + map, O(1)
- [ ] DB 连接池：自定义删除器归还 + wait_for(3s) + 心跳
- [ ] MQ：exclusive=false 多消费者
- [ ] 三高表：AUTO_INCREMENT + ENUM + 软删除
