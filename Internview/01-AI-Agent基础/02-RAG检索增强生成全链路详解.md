# RAG 检索增强生成全链路详解

> 来源：RainCppAI 项目面试追问整理

---

## 一、RAG 的本质

RAG 的本质是**给 LLM 提供外部记忆**。LLM 本身的知识来自训练数据，有截止日期且无法获取私有数据。RAG 通过在推理时实时检索相关文档并注入 Prompt，让 LLM "现学现答"。

核心公式：`最终回答 = LLM(用户问题 + 检索到的相关文档片段)`

---

## 二、完整链路

```
离线阶段（索引构建）：文档 → 分块 → Embedding → 向量存储
在线阶段（查询回答）：用户 query → Embedding → 向量检索 → Top-K → 上下文拼接 → LLM 生成
```

### 2.1 文档分块（Chunking）

| 策略 | 说明 | 优缺点 |
|------|------|--------|
| **固定长度** | 按固定 token 数切分 | 简单，但可能截断语义 |
| **语义分块** | 按段落/句子/章节切分 | 保持语义完整，但块大小不均匀 |
| **滑动窗口** | 固定窗口按步长滑动，相邻块有重叠 | 避免边界截断，但数据量翻倍 |
| **递归分块** | 先按章节，太大再按段落、句子逐级切分 | 业界最佳实践（LangChain RecursiveCharacterTextSplitter） |

### 2.2 Embedding（向量化）

将文本块转化为高维浮点向量（768/1536 维），语义相近的文本向量余弦相似度高。

### 2.3 向量数据库

| 特性 | FAISS | Milvus | ChromaDB |
|------|-------|--------|----------|
| 出品 | Meta | Zilliz | Chroma |
| 语言 | C++ | Go+C++ | Python |
| 部署 | 嵌入进程 | 独立服务 | 嵌入式/独立 |
| 规模 | 1M-1B | 1M-10B+ | <1M |
| 分布式 | 不支持 | 原生支持 | 不支持 |
| 选型 | 单机大规模、C++ 首选 | 生产分布式 | Demo/原型 |

### 2.4 混合检索

| 类型 | 代表 | 原理 | 优势 |
|------|------|------|------|
| **稀疏** | BM25/TF-IDF | 词频统计 | 精确关键词匹配 |
| **稠密** | Embedding 向量 | 语义相似度 | 理解同义词 |

混合检索公式：`最终得分 = α × BM25得分 + (1-α) × 向量相似度得分`

### 2.5 Reranking（重排序）

初步检索 Top-100 → Cross-Encoder（query+doc 拼接过模型）→ 精排 Top-5

- Bi-Encoder（初步）：query 和 doc 分别编码，快但精度有限
- Cross-Encoder（重排）：拼接在一起过模型，精度高但慢

### 2.6 Query 改写

- **扩展**："React hooks" → "React hooks useState useEffect"
- **同义替换**："如何减肥" → "如何减轻体重/如何瘦身"
- **HyDE**：先让 LLM 生成假设答案，用该答案做 Embedding 去检索
- **Multi-Query**：一个问题改写成多个表述，分别检索再合并

### 2.7 项目中的实现

项目使用阿里百炼 RAG 平台，`AliyunRAGStrategy` 调用 API，平台内部完成全链路。

**自建方案（C++）**：
1. ONNX Runtime 加载 BGE-small 做 Embedding
2. FAISS C++ API 构建向量索引
3. 检索 Top-K 后拼接到 Prompt
4. curl 调用 LLM API

## 复习要点速记

- [ ] RAG = 给 LLM 提供外部记忆，推理时实时检索
- [ ] 全链路：分块 → Embedding → 向量存储 → 检索 → 拼接 → 生成
- [ ] 分块策略：固定/语义/滑动窗口/递归（最优）
- [ ] 向量数据库：FAISS(C++单机) / Milvus(分布式) / ChromaDB(原型)
- [ ] 混合检索：BM25(稀疏) + Dense(稠密)，互补
- [ ] Reranking：Cross-Encoder 对粗排结果精排
- [ ] Query 改写：扩展/同义/HyDE/Multi-Query
- [ ] 召回率 = 检索到的相关文档数 / 所有相关文档总数
