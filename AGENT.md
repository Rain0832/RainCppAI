# AGENT.md — AI Agent 开发流程

> 每次 AI Agent 进入 PLAN / IMPLEMENT 模式前，必须首先阅读本文件并遵循流程。

---

## PLAN 阶段强制流程

### ① 阅读文档

阅读以下所有相关文档，并在 PLAN 输出中列出已读文档清单：

- `README.md` — 了解项目架构、模块划分、API 表
- `CONTRIBUTING.md` — 了解依赖规则、头文件管理、提交规范
- `DEVELOP_STANDARD.md` — 了解命名约定、代码风格
- `TODO.md` — 了解当前优先级和待办上下文
- 涉及模块的 `TECHDOC.md` — 了解模块职责、核心文件、耦合边界
- `CHANGELOG.md` — 了解版本历史，确定新版本号和质量基线

### ② 调研对标

- 调研业界主流实现方案（大厂开源项目、标准协议、最佳实践）
- 列出参考来源（GitHub 仓库、RFC、官方文档）
- 分析各方案的优缺点

### ③ 方案给出 / 评估

- 若开发者**未提供**方案：给出 2 个以上可行方案，标注推荐方案
- 若开发者**已提供**方案：对方案做客观评估（优势 / 风险 / 改进建议）
- 方案必须包含：涉及文件清单、核心改动逻辑、测试验证方法

### ④ 设计原则检查

每个方案必须逐条对照以下原则：

| 原则 | 检查点 |
|------|--------|
| **最小侵入** | 是否可避免修改现有接口签名？是否只在模块内部扩展？ |
| **可扩展性** | 新增能力是否通过策略/工厂/插件模式？是否预留扩展点？ |
| **代码可读性** | 命名是否遵循 `DEVELOP_STANDARD.md`？是否有多余嵌套？ |
| **单一职责** | 新增类/函数是否只做一件事？函数是否 ≤ 50 行？ |
| **依赖方向** | 是否违反 `CONTRIBUTING.md` 模块依赖规则？ |
| **错误处理** | 是否覆盖超时、降级、异常路径？ |

---

## IMPLEMENT 阶段

### ⑤ 完成后操作

编译通过后，按顺序执行：

1. **更新 TECHDOC** — 若修改涉及模块的核心文件或流转逻辑，更新对应 `*/TECHDOC.md` 的当前态描述
2. **更新 CHANGELOG** — 在 `CHANGELOG.md` 末尾追加新版本条目（三级递进格式）
3. **必要时更新 README** — 若新增/变更对外 API 或架构层次
4. **必要时更新 TODO** — 若完成某项 P0/P1 待办，勾选或移除

---

## 约束

- 不得在 TECHDOC / README / TODO / DEVELOP_STANDARD / CONTRIBUTING 中保留"旧版本描述"或"改之前是什么样"
- 不得在 TECHDOC 中写入"未来规划"、"预留"、"待迁移"等前瞻性段落
- 只有 `CHANGELOG.md` 保留增量历史，其余文档始终只描述当前态
