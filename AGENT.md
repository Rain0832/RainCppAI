# AGENT.md — AI Agent 开发流程

> 每次 AI Agent 进入 PLAN / IMPLEMENT 模式前，必须首先阅读本文件并遵循流程。

---

## PLAN 阶段强制流程

### ① 阅读文档

阅读以下所有相关文档，并在 PLAN 输出中列出已读文档清单：

- `README.md` — 了解项目架构、模块划分、API 表
- `CONTRIBUTING.md` — 了解依赖规则、头文件管理、提交规范
- `DEVELOP_STANDARD.md` — 了解命名约定、代码风格
- `TODO.md` — 了解当前优先级和待办上下文
- 涉及模块的 `TECHDOC.md` — 了解模块职责、核心文件、耦合边界
- `CHANGELOG.md` — 了解版本历史，确定新版本号和质量基线

### ② 调研对标

- 调研业界主流实现方案（大厂开源项目、标准协议、最佳实践）
- 列出参考来源（GitHub 仓库、RFC、官方文档）
- 分析各方案的优缺点

### ③ 方案给出 / 评估

- 若开发者**未提供**方案：给出 2 个以上可行方案，标注推荐方案
- 若开发者**已提供**方案：对方案做客观评估（优势 / 风险 / 改进建议）
- 方案必须包含：涉及文件清单、核心改动逻辑、测试验证方法

### ④ 设计原则检查

每个方案必须逐条对照以下原则：

| 原则 | 检查点 |
|------|--------|
| **最小侵入** | 是否可避免修改现有接口签名？是否只在模块内部扩展？ |
| **可扩展性** | 新增能力是否通过策略/工厂/插件模式？是否预留扩展点？ |
| **代码可读性** | 命名是否遵循 `DEVELOP_STANDARD.md`？是否有多余嵌套？ |
| **单一职责** | 新增类/函数是否只做一件事？函数是否 ≤ 50 行？ |
| **依赖方向** | 是否违反 `CONTRIBUTING.md` 模块依赖规则？ |
| **错误处理** | 是否覆盖超时、降级、异常路径？ |

## 迭代开发工作流

### 阶段 ①：大 Plan 拆解（用户提出问题 → AI 给出整体方案）

1. AI 阅读 `AGENT.md` 及相关文档，理解项目上下文
2. AI 分析问题根因，形成「大 Plan」：概述问题 + 多个 Small Plan 清单
3. 每个 Small Plan 约 **1 次 Git commit** 的量（5-10 个文件的改动范围）
4. 用户确认大 Plan 后进入逐个小 Plan 迭代

### 阶段 ②：单个 Small Plan 迭代循环

```
用户审批通过
     ↓
┌─────────────────────────────────┐
│ AI 输出 Small Plan N              │  ← 包含：涉及文件、改动逻辑、验证方式
│ （约 1 次 commit 的量）            │
└─────────────────────────────────┘
     ↓ 用户 approve
┌─────────────────────────────────┐
│ AI 在 ACT MODE 实施改动           │  ← 不改版本号、不发版、不做 Git commit
│ 编译通过 + 自测通过               │
└─────────────────────────────────┘
     ↓ 用户 review
┌─────────────────────────────────┐
│ 用户对照 Plan 检查代码质量         │
│ - 是否解决了目标问题？              │
│ - 是否引入回归 bug？              │
│ - 实现是否优雅？                  │
└─────────────────────────────────┘
     ↓ 检查通过，用户指令「commit」
┌─────────────────────────────────┐
│ AI 执行 `git commit`              │  ← 按 CONTRIBUTING.md 格式：
│ （仅 commit，不打 tag）            │    【模块名】type: 描述
└─────────────────────────────────┘
     ↓ 进入 Small Plan N+1，直到大 Plan 全部完成
```

**重要规则**：

- **Commit 权力归用户**：AI 只有在收到用户明确「commit」指令后才能执行 `git commit`
- **发版权力归用户**：AI 只有在收到用户明确「发版」指令后才能更新 `CHANGELOG.md` 或执行 `git tag`
- 如果用户 review 不通过，AI 需要出新的 Small Plan 继续修改，直到满意为止

### 阶段 ③：大 Plan 闭环

所有 Small Plan 迭代完毕后，由用户自行判断：

- 是否合并几个大 Plan 作为一个版本
- 版本号如何递增（major / minor / patch）
- 是否需要执行发版工作流（更新 CHANGELOG / TECHDOC / 打 Tag）

AI 在收到用户明确「发版」指令后，读取 `/root/.agents/skills/release_workflow` 并执行标准化发版流程。

### 阶段 ④：AI 行为约束

| ❌ 禁止 | ✅ 正确做法 |
|---------|-------------|
| 未收到「commit」指令时执行 `git commit` | 等待用户说「commit」或「提交」 |
| 未收到「发版」指令时更新 `CHANGELOG.md` | 等待用户说「发版」或「更新版本」 |
| 自动修改版本号 | 版本号完全由用户决定 |
| Small Plan 改动中执行 `git tag` | Tag 仅由发版流程产生 |
| 改动完成后自动推送到远程 | 推送由用户自行操作 |

**允许的操作**（在 Small Plan 实施时）：

- 改动涉及的模块如果核心逻辑有变更，在 commit 同时更新对应的 `TECHDOC.md`
- 改动涉及的 `TODO.md` 条目完成时，在 commit 同时勾选

---

## 约束

- 不得在 TECHDOC / README / TODO / DEVELOP_STANDARD / CONTRIBUTING 中保留"旧版本描述"或"改之前是什么样"
- 不得在 TECHDOC 中写入"未来规划"、"预留"、"待迁移"等前瞻性段落
- 只有 `CHANGELOG.md` 保留增量历史，其余文档始终只描述当前态
